# ADR 001: Модульный монолит

**Status:** Accepted
**Date:** 2026-04-14

## Context

RESERV.atiion — букинг-платформа с 4 bounded contexts (Identity, Catalog, Booking, Payment). Нужно выбрать между:
1. Монолит без разделения на модули (классический Laravel MVC)
2. Модульный монолит (одно приложение, строгие границы)
3. Микросервисы (N сервисов, общение по сети)

## Decision

**Модульный монолит.** Каждый BC — отдельный модуль в `app/Modules/{Name}/`. Межмодульное общение через Domain Events или явные Application Service interfaces в Shared.

## Consequences

**Плюсы:**
- Чёткие границы в коде (сохраняет DDD-дисциплину)
- Один деплой (простота операций)
- Рефакторинг внутри модуля безопасен
- Легко извлечь модуль в отдельный сервис, когда упрётся в масштаб

**Минусы:**
- Чуть больше boilerplate по сравнению с простым Laravel
- Требуется дисциплина — не поддаться соблазну прямых вызовов между модулями

**Компромисс:** Sharing БД — одна PostgreSQL на все модули. Нет физической изоляции данных. Если модуль выделится в сервис — придётся мигрировать таблицы.

## Alternatives considered

- **Классический Laravel MVC** — отвергнут: нет границ между BC, всё переплетётся через 6 месяцев
- **Микросервисы** — отвергнут: преждевременная оптимизация, огромный overhead на ops, deploy, observability для команды из одного человека
